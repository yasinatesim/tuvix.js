Docs / Introduction

What is Tuvix.js?

Tuvix.js is a lightweight, framework-agnostic microfrontend orchestrator. It lets you build large-scale frontend applications as a collection of small, independently deployable apps — each owned by a different team, each using its own framework.

The problem it solves

As frontend applications grow, monolithic SPAs become painful to maintain and scale. Multiple teams working on the same codebase lead to slow builds, complicated deployments, and tightly coupled code.

Microfrontend architecture solves this by letting each team ship their piece of the UI independently. But without the right orchestrator, microfrontends become an infrastructure headache. Tuvix.js handles the hard parts: loading, routing, communication, and lifecycle management.

Core design principles

  • Framework agnostic — React, Vue, Svelte, Angular, or plain JS all work as micro apps.
  • Zero runtime deps — The entire framework has zero external dependencies.
  • TypeScript first — Written entirely in TypeScript with full type exports.
  • Modular packages — Install just what you need via scoped packages.
  • Predictable lifecycle — Every micro app follows a clear bootstrap → mount → unmount flow.

Getting Started/Installation

Installation

Tuvix.js is distributed on npm as a monorepo. You can install the umbrella package for convenience, or individual scoped packages for fine-grained control.

Option A — Umbrella package

Install everything in one go. Best for getting started quickly.

npm install tuvix.js
# or
pnpm add tuvix.js
# or
yarn add tuvix.js

Option B — Individual packages

Install only what your project needs. Better for tree-shaking and smaller bundles.

# Core orchestrator (required)
npm install @tuvix.js/core

# Add routing
npm install @tuvix.js/router

# Add inter-app communication
npm install @tuvix.js/event-bus

# Add dynamic loading
npm install @tuvix.js/loader

Requirements

RequirementVersion
Node.js≥ 18.0.0
TypeScript (optional)≥ 5.0
BrowserModern browsers (ES2020+)
Tip Tuvix.js is ESM-first. Make sure your bundler supports exports field resolution (Vite, Webpack 5, Rollup, esbuild).

Getting Started/Quick Start Guide

Quick Start

Build your first microfrontend shell in under 10 minutes. This guide walks through creating a host application that loads two micro apps based on the current URL.

1. Project structure

my-project/
├── shell/            ← Host application
│   ├── index.html
│   └── main.ts
├── app-dashboard/    ← Micro app 1
│   └── main.ts
└── app-settings/     ← Micro app 2
    └── main.ts

2. Create the HTML shell

<!-- shell/index.html -->
<html>
<body>
  <div id="header"></div>
  <nav>
    <a href="/dashboard">Dashboard</a>
    <a href="/settings">Settings</a>
  </nav>
  <main id="main"></main>
</body>
</html>

3. Configure the orchestrator

// shell/main.ts
import { createOrchestrator } from 'tuvix.js';

const orchestrator = createOrchestrator({
  router: {
    mode: 'history',
    routes: [
      { path: '/dashboard/*', app: 'dashboard' },
      { path: '/settings/*',  app: 'settings'  },
    ],
  },
  onError(error, appName) {
    console.error(`App "${appName}" failed:`, error);
  },
});

orchestrator.register({
  name:       'dashboard',
  entry:      'https://cdn.example.com/dashboard/main.js',
  container:  '#main',
  activeWhen: '/dashboard/*',
});

orchestrator.register({
  name:       'settings',
  entry:      'https://cdn.example.com/settings/main.js',
  container:  '#main',
  activeWhen: '/settings/*',
});

orchestrator.start();

4. Create a micro app

// app-dashboard/main.ts
import { defineMicroApp } from 'tuvix.js';

export default defineMicroApp({
  name: 'dashboard',

  async bootstrap() {
    // One-time setup — called once per page load
    console.log('Dashboard bootstrapped');
  },

  async mount({ container, props }) {
    // Render your app into the container
    container.innerHTML = '<h1>Dashboard</h1>';
  },

  async unmount({ container }) {
    // Clean up DOM, listeners, timers
    container.innerHTML = '';
  },
});
That's it! Navigate to /dashboard and Tuvix.js will automatically fetch, bootstrap, and mount the dashboard micro app.

Core Concepts/Orchestrator

Orchestrator

The Orchestrator is the central coordinator of Tuvix.js. It manages app registration, lifecycle transitions, route-based activation, and error handling.

Creating an orchestrator

Use createOrchestrator() to instantiate the orchestrator with your configuration.

import { createOrchestrator } from '@tuvix.js/core';

const orchestrator = createOrchestrator({
  router: {
    mode: 'history',
    routes: [/* ... */],
  },
  prefetch: { strategy: 'idle' },
  onError(error, appName) { /* ... */ },
  onStatusChange(appName, status) { /* ... */ },
});

Registering micro apps

Register micro apps before calling start(). Each app needs a name, an entry URL, and a container selector.

orchestrator.register({
  name:       'dashboard',         // unique identifier
  entry:      'https://cdn.../main.js', // script URL
  container:  '#main-content',     // mount target
  activeWhen: '/dashboard/*',       // URL pattern
  props: { theme: 'dark' },          // optional props
});

The activeWhen option

ValueBehavior
'/'Always active (navbar, header)
'/dashboard/*'Active at /dashboard and all sub-paths
'/settings'Active only at exact /settings
(path) => booleanCustom activation function
undefinedManual control — never auto-activates

Prefetch strategies

StrategyDescription
'none'No prefetching (default)
'immediate'Prefetch all apps at startup
'idle'Prefetch during browser idle time
'hover'Prefetch on user interaction (planned)
Core Concepts/Lifecycle

Lifecycle

Every micro app in Tuvix.js follows a predictable lifecycle. Understanding these states helps you manage resources, avoid memory leaks, and debug issues.

Lifecycle states

registered → bootstrapping → bootstrapped → mounting → mounted
                                                              ↓
                                             unmounting ← updating
                                                  ↓
                                             unmounted
StateDescription
registeredApp config stored, nothing loaded yet
bootstrappingScript is being fetched and bootstrapped
bootstrappedModule loaded, bootstrap() called
mountingmount() is executing
mountedApp is fully rendered and interactive
updatingProps changed, update() is executing
unmountingunmount() is executing
unmountedApp cleaned up, container empty
errorAn error occurred in any lifecycle method

Lifecycle hooks

defineMicroApp({
  name: 'my-app',

  /**
   * Called ONCE when the module is first loaded.
   * Use for one-time setup: auth tokens, translations, etc.
   * Not called again on subsequent mounts.
   */
  async bootstrap() {
    await initializeSdk();
  },

  /**
   * Called every time the app should render.
   * Receive container and props from the orchestrator.
   */
  async mount({ container, props }) {
    app = createApp(AppComponent);
    app.mount(container);
  },

  /**
   * Called when props update while the app is mounted.
   * Optional — omit if you don't need prop updates.
   */
  async update({ props }) {
    store.dispatch(setProps(props));
  },

  /**
   * Called every time the app should unmount.
   * ALWAYS clean up — remove listeners, destroy instances.
   */
  async unmount({ container }) {
    app.unmount();
    container.innerHTML = '';
  },
});
Important Always clean up in unmount(). Remove event listeners, destroy framework instances, and clear the container. Failure to do so causes memory leaks.
Core Concepts/Routing

Routing

Tuvix.js has a built-in router that maps URL patterns to micro app activation. It supports history and hash modes, URL parameters, wildcards, and navigation guards.

Router modes

createOrchestrator({
  router: {
    mode: 'history', // uses pushState — needs server config
    // or
    mode: 'hash',    // uses #hash — works everywhere
    routes: [/* ... */],
  },
});

Route patterns

PatternMatches
/dashboardExactly /dashboard
/dashboard/*/dashboard, /dashboard/stats, /dashboard/user/42
/users/:id/users/42 — captures id param
/Always matches — use for persistent apps

Navigation guards

const router = orchestrator.getRouter();

router.beforeEach(async ({ from, to }) => {
  const isLoggedIn = await checkAuth();
  if (!isLoggedIn && to !== '/login') {
    router.replace('/login');
    return false; // cancel navigation
  }
  return true;
});

Programmatic navigation

const router = orchestrator.getRouter();

router.push('/dashboard');      // adds history entry
router.replace('/dashboard');   // replaces current
router.back();                  // browser back

// Or use the shorthand on orchestrator
orchestrator.navigateTo('/settings');
Core Concepts/Event Bus

Event Bus

The event bus is a type-safe pub/sub system for communication between micro apps. It keeps apps decoupled while still allowing them to share state and trigger actions across boundaries.

Getting the event bus

// Option A: Get from orchestrator instance
const bus = orchestrator.getEventBus();

// Option B: Global singleton (shared across all apps)
import { getGlobalBus } from 'tuvix.js';
const bus = getGlobalBus();

// Option C: Create an isolated instance
import { createEventBus } from 'tuvix.js';
const bus = createEventBus({ debug: true });

Subscribe and emit

// Subscribe
const unsub = bus.on('user:login', (data) => {
  console.log(`${data.name} logged in`);
});

// Emit
bus.emit('user:login', { userId: 42, name: 'Ahmet' });

// One-time listener
bus.once('app:init', (config) => {
  configure(config);
});

// Wildcard — listen to all events
bus.onAny((eventName, data) => {
  console.log(`[${eventName}]`, data);
});

// Clean up
unsub(); // or bus.off('user:login', handler)

Recommended event naming

Use a domain:action convention to avoid collisions:

'user:login'        // ✓ domain:action
'user:logout'
'cart:item-added'
'theme:changed'
'tuvix:app:mounted' // ✓ framework internal prefix
Core Concepts/Module Loading

Module Loading

Tuvix.js dynamically loads micro app scripts at runtime. The loader handles caching, retries, timeouts, and prefetching automatically.

How micro apps expose themselves

For the loader to find a micro app's exports, the app must register itself in the global module registry:

// Tuvix.js handles this automatically when you use defineMicroApp:
defineMicroApp({ name: 'my-app', /* ... */ });
// ↑ internally sets window.__TUVIX_MODULES__['my-app'] = module

// For manual registration:
window.__TUVIX_MODULES__ = window.__TUVIX_MODULES__ || {};
window.__TUVIX_MODULES__['my-app'] = {
  mount({ container }) { /* ... */ },
  unmount({ container }) { /* ... */ },
};

Entry formats

// Simple: single script URL
entry: 'https://cdn.example.com/app/main.js'

// Full config: scripts + styles + html
entry: {
  scripts: ['https://cdn.example.com/app/main.js'],
  styles:  ['https://cdn.example.com/app/style.css'],
}

Loader configuration

// Advanced: use ModuleLoader directly
import { createLoader } from '@tuvix.js/loader';

const loader = createLoader({
  timeout:    15000,  // 15s timeout per load
  retries:    3,      // retry 3 times on failure
  retryDelay: 1000,  // 1s between retries
});

const result = await loader.load('dashboard', entry);
if (result.module) {
  await result.module.mount({ container });
}
API Reference/@tuvix.js/core

@tuvix.js/core

The core package exports the Orchestrator, createOrchestrator, defineMicroApp, and related types.

Functions

createOrchestrator(config?) function

Creates and returns a new Orchestrator instance.

createOrchestrator(config?: OrchestratorConfig): Orchestrator
defineMicroApp(definition) function

Defines and registers a micro app module. Validates lifecycle methods and registers the module in window.__TUVIX_MODULES__ for the loader.

defineMicroApp(definition: MicroAppDefinition): MicroAppModule

Orchestrator class

orchestrator.register(config) method

Register a micro app. Must be called before start().

orchestrator.start() method → Promise<void>

Start the orchestrator. Sets up router listener and mounts active apps for the current URL.

orchestrator.mountApp(name) method → Promise<void>

Manually mount a registered app. Use when activeWhen is not set.

orchestrator.unmountApp(name) method → Promise<void>

Manually unmount a mounted app.

orchestrator.getEventBus() method → IEventBus

Returns the shared event bus instance for this orchestrator.

orchestrator.getRouter() method → IRouter | null

Returns the router instance, or null if no router was configured.

orchestrator.destroy() method → Promise<void>

Unmounts all apps, removes all listeners, and destroys the orchestrator. Cannot be reused after this call.

Guides/React Integration

React Integration

Mount React applications as micro apps in Tuvix.js with full root management and cleanup.

Mounting a React app

import { defineMicroApp } from 'tuvix.js';
import { createRoot } from 'react-dom/client';
import App from './App';

let root: any = null;

export default defineMicroApp({
  name: 'dashboard',

  async mount({ container, props }) {
    root = createRoot(container);
    root.render(<App {...props} />);
  },

  async update({ props }) {
    root.render(<App {...props} />);
  },

  async unmount({ container }) {
    root?.unmount();
    root = null;
  },
});
React 18+ Always use createRoot from React 18, not ReactDOM.render. And always call root.unmount() in the unmount hook to prevent memory leaks.
Guides/Vue Integration

Vue Integration

Mount Vue 3 applications as micro apps with the Composition API and full lifecycle support.

import { defineMicroApp } from 'tuvix.js';
import { createApp } from 'vue';
import App from './App.vue';

let app: any = null;

export default defineMicroApp({
  name: 'settings',

  async mount({ container, props }) {
    app = createApp(App, props);
    app.mount(container);
  },

  async update({ props }) {
    // Update via provide/inject or Pinia store
    app.config.globalProperties.$props = props;
  },

  async unmount({ container }) {
    app?.unmount();
    app = null;
    container.innerHTML = '';
  },
});

  name:  'dashboard',

Guides/Migrating to Tuvix

Migrating to Tuvix.js

Migrating an existing SPA to Tuvix.js can be done incrementally — one route at a time.

Incremental migration strategy

1

Create the shell

Extract your existing app's layout (header, nav, footer) into a shell app that always mounts.

2

Extract one route

Move one feature (e.g., the Settings page) into a separate micro app. Deploy it independently.

3

Register and verify

Register the new micro app in the shell and verify it loads and unmounts correctly.

4

Repeat for each route

Continue extracting routes as micro apps. Your original app shrinks; micro apps grow.

No big bang required You can run Tuvix.js alongside your existing monolith. Start with just one micro app and migrate at your own pace.

Packages/Sandbox

@tuvix.js/sandbox

The sandbox package provides CSS and JavaScript isolation for micro apps — preventing style leaks and global variable collisions.

Features

  • Shadow DOM CSS isolation — Scoped styles that don't leak between micro apps
  • Proxy-based JS sandbox — Isolated global scope per app
  • Easy opt-in per app — Enable sandbox only where needed

Usage

import { createSandbox } from '@tuvix.js/sandbox';

const sandbox = createSandbox({ css: true, js: true });
const shadowRoot = sandbox.activate(container);

// App runs in isolation — styles and globals are scoped

sandbox.deactivate(container);
